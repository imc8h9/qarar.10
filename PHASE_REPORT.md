## ⭐⭐ مقارنة السيناريو — جوهر «قرار» صار مرئيًا

المحرك كان **يحسب** هذه الأرقام منذ البداية — **ولم تُعرض أبدًا.**

```
┌──────────────┐   ┌─────────┐   ┌──────────────┐
│ بدون القرار   │ ← │ −٣٬٠٠٠  │ ← │  مع القرار    │
│   ٣٬٠٧٠      │   └─────────┘   │     ٧٠       │
│ ✓ فوق الحد   │                 │ ⚠️ تحت الحد   │
└──────────────┘                 └──────────────┘
```

**هذا هو المشروع كله في صف واحد.** الفرق بين «أين صرفتُ مالي؟»
و«ماذا سيحدث لو اتخذتُ هذا القرار؟» — مرئي في ثانية.

🔴 **أرقام من `r.comparison` مباشرة — لا حساب جديد. المحرك لم يُلمَس.**

```
المستوى ٥  إجراءات سريعة (٤)                       ← البدء
```

الجملة تتغيّر حسب حالتك الفعلية (مستقر / يحتاج انتباهًا / حرج) — **من المحرك**.

**الحالة:** ✅ **مُصلَح — 43/43 اختبارًا بالنقر · 500/500 إجمالًا.**

---

## الخلاصة في ثلاثة أسطر

1. **المحرك كان يعمل** — الأرقام تُحسب وتُرسم بشكل صحيح.
2. **لكن ردّ الفعل البصري اختفى** — لا شاشة تحميل، ولا رأس نتيجة،
   والصفحة تقفز إلى منتصف كتلة طويلة بلا عنوان.
3. **من منظور المستخدم = الزر ميت.** وهذا تعريف صحيح.

---

## ⭐ السبب الجذري

في **Startup Hardening (v2.2)** وضعتُ:

```js
ld.style.display = 'none';     // 🔴 inline style
```

و`showLoading()` كان يُزيل الـCSS class فقط.
**الـinline style يتفوّق على الـclass** ⇒ `showLoading()` صار **بلا أثر**.

⇒ **شاشة التحميل المميزة اختفت من المنصة كلها** — التحليل *والتنقل*.

**الإثبات:** عيّنتُ `display` كل 40ms بعد الضغط →
`none,none,none,...` × 26 — **لم تظهر ولو لإطار واحد.**

**الإصلاح:**
```js
el.style.display = '';         // امسح الـinline style أولًا
el.classList.remove('hide');
```

## 1. ⭐ نعم — كانت مشكلة Store Subscription Lifecycle بالضبط

**فرضيتك كانت صحيحة تمامًا.** وأثبتُّها بالنقر، لا بالافتراض.

### الكود المعطوب

```js
/* عند تحليل السكربت — المتجر «أ» */
QararStore.subscribe(function(){ refreshCurrentView(); });

/* لاحقًا — prepareDependencies() يستبدل المتجر */
window.QararStore = QararFallbackStore;      // ← المتجر «ب»
```

**التسلسل الكارثي:**
1. الواجهة تشترك في المتجر **«أ»**
2. `prepareDependencies()` يستبدل `window.QararStore` بـ**«ب»**
3. المستخدم يضيف عملية → تُحفظ في **«ب»**
4. **«ب»** يُنادي مشتركيه — **لكن قائمته فارغة**
5. الواجهة مشتركة في **«أ»** الذي لا يستخدمه أحد
6. **البيانات تتغير · الشاشة لا تتحرك**

### الإثبات (بالنقر، في متصفح حقيقي)

```
=== على المتجر البديل ===
A) إضافة مصروف 200:
     المتجر : 400000   ✓ (تغيّر)
     الشاشة : ٤٬٢٠٠     🔴 لم تتغير!

C) تعديل الكهرباء إلى 900:
     المتجر : 90000    ✓ (تغيّر)
     الشاشة : لا تعرض ٩٠٠  🔴
```

**هذا بالضبط ما رأيتَه:** «أضفت عملية والرصيد لم يتغير» · «عدّلت الكهرباء ولم تُحفظ».

## 2. هل كان التطبيق يستخدم Fallback Store في الوضع الطبيعي؟

**لا — والحمد لله.** في الوضع الطبيعي (HTTP أو file:// بعد Extract All):

```
QararStore.__isFallback : false ✓
مستوى الاسترداد          : 1 ✓
```

**لكن** حالما ينتقل إلى المتجر البديل (فتح من ZIP · localStorage محظور · فشل ملف)،
**تنقطع الواجهة عن المتجر تمامًا** — وهذا ما حدث عندك.

**والأسوأ:** لم تكن هناك أي رسالة خطأ. **فشل صامت.**

## 3. سبب تعطل «اختبار القرار»

سببان:

**أ. الاعتماد على `setTimeout`:**
```js
showLoading('...');
setTimeout(() => { hideLoading(); runDecisionAnalysis(); }, 550);
```
لو رمى `runDecisionAnalysis` بعد `hideLoading`، يبقى المستخدم **بلا نتيجة وبلا خطأ**.

**ب. لا `try/catch/finally`:** أي خطأ ⇒ زر معطّل للأبد + لا رسالة.

## 4. ما الذي أصلحته

### أ. ⭐ `bindActiveStore()` — دورة حياة صحيحة

```js
function bindActiveStore(){
  if (subscribedStore === window.QararStore) return;   // لا تكرار

  if (activeStoreUnsubscribe) activeStoreUnsubscribe(); // فُك القديم

  subscribedStore = window.QararStore;
  activeStoreUnsubscribe = subscribedStore.subscribe(handleFinancialStateChange);
}
```

**تُستدعى في 4 مواضع:** بعد `prepareDependencies()` في التمهيد · في `enterAppSafely()`
**قبل الرسم** · بعد أي تحويل للمتجر البديل · قبل أي تعديل مالي.

### ب. ⭐ `commitFinancialMutation()` — النقطة المركزية الوحيدة

**لا نعتمد على Subscription وحدها — فهي هشّة.**

```js
function commitFinancialMutation(operation, options){
  bindActiveStore();                          // ① تأكد من الارتباط
  const result = operation();                 // ② نفّذ
  QararStore.recalculateFinancialSnapshot();  // ③ أعد الحساب
  refreshCurrentView();                       // ④ 🔴 أعد الرسم صراحةً
  toast(successMessage);                      // ⑤ رسالة نجاح
  // ⑥ catch → رسالة خطأ واضحة
}
```

**كل تعديل مالي يمر من هنا:** إضافة/تعديل/حذف عملية · فاتورة · التزام · ملف مالي · قرار.

### ج. تحليل القرار — `try/catch/finally` بلا `setTimeout`

```js
try{
  button.disabled = true; button.textContent = 'جارٍ التحليل...';
  bindActiveStore();
  const result = QararEngine.analyzeDecision(input);   // ⭐ فوري — لا انتظار
  renderDecisionResult(result);
}catch(err){
  showDecisionError(err);        // رسالة + إعادة محاولة + إعادة ضبط
}finally{
  button.disabled = false;       // 🔴 الزر لا يموت أبدًا
}
```

### د. حالة التخزين — صريحة بلا تضليل

```
محرك القرار : يعمل ✓
التخزين     : محلي دائم ✓   أو   وضع استرداد ⚠️
المساعد     : وضع احتياطي ⚠️
البيئة      : Sandbox تجريبية
```

**لا يقول التطبيق «حُفظ» وهو على الذاكرة.** لو كان على المتجر البديل:
> ⚠️ وضع استرداد — التغييرات مؤقتة ولن تبقى بعد إغلاق الصفحة.

# 🔴 HOTFIX 2.2 — Loading الأبدي (Startup Reliability)

**الخطورة:** P0 — المستخدم يبقى عالقًا في شاشة التحميل ولا يدخل المنصة.
**الحالة:** ✅ **مُصلَح ومُحصَّن — 53 اختبار بدء في متصفح حقيقي.**

---

## 1. هل كانت المشكلة بسبب Runtime عندك أم طريقة فتح الملفات؟

**لا هذا ولا ذاك. كانت عيبًا حقيقيًا في الكود — وأنت كنت محقًا تمامًا في رفض تفسير «نجح عندي».**

### السبب الجذري

```html
<!-- قبل -->
<div id="loading" class="loading">
```

```css
.loading{ position:fixed; inset:0; z-index:100; }   /* ← طبقة حاجبة كاملة */
```

**شاشة التحميل كانت تبدأ ظاهرة في HTML**، ولا تُخفى إلا بسطر JavaScript
في **نهاية** سكربت التمهيد:

```js
hideLoading();   // ← آخر سطر
```

**النتيجة:** أي فشل في **أي سطر قبله** ⇒ لا يُنفَّذ ⇒ **الطبقة الحاجبة تبقى للأبد.**

### ولماذا نجح عندي وفشل عندك؟

لأن الفشل كان يعتمد على **البيئة**، لا على الكود وحده. ثلاثة مسارات:

| المسار | ما يحدث |
|---|---|
| **١. سطر خارجي في المستوى الأعلى** | `const Money = QararEngine.Money;` — لو تأخر/فشل تحميل `decision-engine.js` → `ReferenceError` → **يتوقف السكربت كله** → `hideLoading()` لا يُنفَّذ |
| **٢. localStorage محظور** | الوضع الخاص/سياسات المتصفح → `QararStore` يرمي → `views.home()` يفشل |
| **٣. ملف JS لا يُحمَّل** | بطء قرص · كاش · قيود `file://` → نفس النتيجة |

عندي كانت الملفات تُحمَّل فورًا من قرص محلي سريع. **عندك — لأي سبب — لم تكتمل.**
هذا بالضبط **هشاشة تصميم**، لا «مشكلة بيئة».

**الاستنتاج:** التصميم كان يفترض أن *كل شيء ينجح*. الآن يفترض أن *أي شيء قد يفشل.*

## 2. كيف أصبحت شاشة البداية Fail-safe؟

### التغيير الجوهري — عكس الافتراض

```html
<!-- بعد -->

```

```css
.splash.hide, .loading.hide{ display:none !important; }   /* لا يمكن تجاوزها */
```

**الآن:** شاشة التحميل **لا تظهر إلا عند استدعاء `showLoading()` صراحةً.**
فشل أي كود لا يُظهرها — لأنها مخفية أصلًا.

### ⭐ `enterAppSafely()` — المصدر الوحيد للدخول

كل مسارات الدخول تمرّ من هنا. **لا تكرار للمنطق في أي مكان.**

```js
function enterAppSafely(reason){
  if (window.__qararEntered) return;      // idempotent
  window.__qararEntered = true;
  clearTimeout(window.__watchdogTimer);

  try{ splash.classList.add('hide'); splash.style.display='none'; }catch(e){}
  try{ loading.classList.add('hide'); loading.style.display='none'; }catch(e){}

  try{ view.innerHTML = views.home(); }catch(e){ /* بديل نصي */ }

```

**كل خطوة معزولة في `try/catch`** — فشل خطوة لا يمنع البقية.
حتى لو فشلت كل الخطوات، تُزال الطبقات الحاجبة **قسرًا** بـ`style.display='none'`.

### تسلسل الزر — حتمي (A → G)


- تعديل · حذف (بتأكيد) · **منع الضغط المزدوج**.

### الملاحظة 6 — Transaction ≠ Planned Decision
- القرار المخطط **لا يخصم** (اختبار `§9`).
- إجراء منفصل **«تم تنفيذ القرار»** → يتحول إلى Transaction ويُخصم **مرة واحدة**.
- محاولة تنفيذ ثانية → `ALREADY_EXECUTED` (اختبار `§10`).

### الملاحظة 7 — مساعد حقيقي
انتقل من keyword matching إلى **AI Agent فعلي** — التفاصيل في القسم 5.

### منع الحساب المزدوج (موثّق ومُختبر)
| القاعدة | التنفيذ |
|---|---|
| التزام/فاتورة **مدفوعة** | `settled=true` → **تخرج من التوقعات** |
| قرار **منفَّذ** | `status='executed'` → لا يُحسب كتوقع |
| **حذف** العملية المرتبطة | يفك التسوية → الحدث **يعود** للتوقعات |
| نفس شهر الفاتورة | **تحديث** لا تكرار |

اختبار `§12`: دفع الإنترنت فعليًا → الالتزامات تنزل من 3 إلى 2،
والرصيد ينقص 300 **مرة واحدة**، وأقل رصيد **يبقى 70** (لا 70−300).

## 6. Fallback — الأهم للعرض

**تعطل الذكاء الاصطناعي ≠ تعطل المنصة.**

عند فشل المزود (أو غياب المفتاح، أو تشغيل الملف محليًا):
1. لافتة **برتقالية صريحة**: «الوضع الاحتياطي المحدود».
2. المساعد يعمل بقواعد نصية بسيطة.
3. **⭐ الأرقام تبقى صحيحة تمامًا** — لأن المحرك يعمل في المتصفح.

اختبار `§7`: مع تعطيل المزود، المساعد **ما زال يعطي 70 و1,570**.

**هذا يعني: حتى لو انقطع الإنترنت في قاعة التحكيم، عرضك يعمل.**
